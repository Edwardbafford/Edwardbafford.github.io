Handling Errors in Web Applications

Here is the pattern I currently use to handle exceptions within my web applications. This applies mainly to languages with checked exceptions such as Java, although I think it could still be applied to, for example, a Python application.


Controllers

Controllers are the sections of code which act as the inferface between the client and the rest of the system. Because of this controllers should handle any exception bubbles up from the application and return an appropriate response to the user. I prefer to handle this in an aspect oriented way to avoid boilerplate code.


Services

Services are the layer below controllers and handle the business logic of the different requests. They delegate tasks to more fundemental components and coordinate appropriate responses.

Becuase services are closely tied to the user stories the service layer should return application related exceptions by catching lower level exceptions coming from the lower level processing components. For example, if a request to an authentication service fails the service layer should catch this error and throw a new exception identifying that the authentication step failed while embedding the exception that caused it. By layering exceptions in this way information about the exception is not lost while still providing the user with an understandable message.

Exceptions returned from the service layer should be recognizable by the end user.


Core Components

The "core components" are just lower level, reusable sections of the code that can be leveraged during many different services calls. These can be, for example, a persistence layer, external services, other libraries, etc.

Often times these core components are outside of the users control but usually represent a general issue that occurred during processing. Examples include parsing and connection exceptions. They help give a general idea of the problem. For example, instead of seeing a null pointer exception a parsing exception gives the person troubleshooting a better idea of the issue and how to approach it.

Exceptions returned from lower level components should be recognizable by developers familiar with the application.
