Application Database Design

Here I discussed some best practices for setting a database with an enterprise web application.

First what kind of databases management system should you use?

SQL DBMS

For most use cases and as a default choice I recommend a SQL DBMS. Why?

  • Provides structure to your system
  • Integrates easily into applications with ORM frameworks
  • Comes with fancy fleshed out features
    • Transaction management
    • Prepared statements
    • Internally computed functions such as joins and aggregations
    • Stored procedures

Overall I think they are conducive to efficiently building robust applications and almost every web application should use one for at least part of their database.

Document DBMS

Document DBMS can be useful as a supplimental system.

I think there are two main reasons for storing parts of your database in a Document DBMS

  • Bulky data/lots of data
  • Messy or naturally disorganized data

All those "fancy and fleshed out features" come at the cost of scalability in a SQL DBMS. If you need to store large chunks of text or many many entries of something it is probably better to store it in a Document DBMS. This is simply because Document DBMS can scale indefinitely as clusters where as SQL DBMS cannot.

As for messy data, Document DBMS do have defined schemas. So if the properties of an object vary and/or are unpredictible it can be worth it to just throw it into a Document DBMS collection and manage it within the application.

General Tips

As I've already implied, I am a fan of a hybrid approach. Start off with a SQL DBMS and suppliment with a Document DBMS as needed based on the strengths I've defined above.

Detach bulky sections of an object. This idea kind of goes along with the hybrid approach, if a certain database object has some lightweight fields along with a bulky field it is usually best to split them up into two independent objects (maybe in two different DBMS!). Essentially this just makes it easier to choose when you want to pull and update the bulky part of the object in your application instead of constantly having to worry about using up resources by unnecissarily working with it.

Unless it causes a lot of resource overhead make sure to define all objects you want to use in the application in the database schema. You don't want to have to manually create and manage a composite object in your application when your DBMS and an ORM framework can do the same thing much more efficiently.
